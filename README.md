# Application of Artificial Intelligence Methods in the Requirements Engineering Management Process

**Master's Thesis — Vilnius Gediminas Technical University, 2025**

---

## Overview

This project implements a three-stage AI pipeline that transforms raw, ambiguous stakeholder requirements into a structured, classified, and traceable knowledge graph. The system addresses three core challenges in software requirements engineering: requirement ambiguity and incompleteness, classification inconsistency between functional and non-functional requirements, and poor visibility of dependencies and conflicts across a requirements set.

---

## Pipeline Stages

### Stage 1 — Intelligent Requirement Structuring

Raw requirement text submitted by stakeholders is processed by a locally deployed Large Language Model (Qwen2.5-7B-Instruct) using constrained prompting. The model extracts structured fields including actors, actions, objects, and constraints, and explicitly identifies missing or vague information. A weighted completeness score is calculated across three components: field presence, vague term detection, and missing information density. Requirements scoring below 0.80 are flagged as incomplete and the system automatically generates targeted clarification questions specific to the identified gaps. Requirements scoring above the threshold proceed to Stage 2.

### Stage 2 — Multi-Model Consensus Classification

Each structured requirement is independently classified by three methods running in parallel:

- **NoRBERT** — a BERT-based transformer model fine-tuned on the PROMISE NFR corpus for binary functional/non-functional classification
- **LLM Classifier** — Qwen2.5-7B-Instruct with a detailed prompt specifying NFR category signal terms and a critical rule for quantitative constraints
- **Rule-Based Classifier** — deterministic keyword matching across eight NFR subcategories loaded from a YAML configuration file

A consensus voting mechanism combines the three outputs. When all three agree the requirement receives HIGH confidence. When two agree it receives MEDIUM confidence and proceeds automatically. When none agree it is routed to a manual review queue. This multi-method approach ensures that no single classifier's bias dominates the final classification output.

### Stage 3 — Graph-Based Traceability Management

Classified requirements are stored as nodes in a Neo4j graph database. Each node carries the original text, classification label, confidence score, and a 384-dimensional sentence embedding generated by the all-MiniLM-L6-v2 model. Dependency and conflict detection follows an LLM-first architecture: the language model analyses each requirement pair for logical dependencies and semantic conflicts in a single inference call, with semantic similarity and keyword-based methods providing corroborating evidence. Relationships validated through this process are committed to the graph as typed edges — DEPENDS_ON and CONFLICTS_WITH — with metadata recording the detection methods and confidence scores. The resulting knowledge graph supports dependency chain traversal, conflict identification, and impact analysis across the full requirements set.

## Technology Stack

| Component | Technology |
|---|---|
| Language | Python 3.11 |
| LLM | Qwen2.5-7B-Instruct (4-bit NF4 quantization) |
| LLM Framework | HuggingFace Transformers + BitsAndBytes |
| RE Classifier | NoRBERT (BERT fine-tuned on PROMISE NFR) |
| Sentence Embeddings | all-MiniLM-L6-v2 (sentence-transformers) |
| Graph Database | Neo4j Community Edition 5.13 |
| Validation | Pydantic v2 |
| NLP Utilities | NLTK |
| Runtime | Google Colab (T4 GPU) |
